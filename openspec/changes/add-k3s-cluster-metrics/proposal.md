## Why

After slice 20, the obs cluster's Loki carries pod logs end-to-end and the spine — gateway collector → cross-cluster mTLS → obs collector → LGTM stack — is fully built. The metrics leg is the last of the three signal types still missing a cluster-side feeder: backend application metrics already arrive via OTLP push (slices 6 + 18c), but **the cluster itself** — node CPU/memory/disk/load, per-pod CPU/memory by namespace, deployment / replica / pod-phase / restart state — has no producer dialing the gateway. README §"Forward arc" line 459 names this exact slice (slice 21, `add-k3s-cluster-metrics`) as the moment the cluster control plane and node plane join the same transport spine, closing the data-plane gap for the third and final signal type before slice 22 retires the compose mirror.

Doing it now keeps the agent/gateway pattern slice 20 just introduced amortized across two slices instead of refactored mid-stream: the in-cluster scrape happens on a node-local agent, the cross-cluster TLS / fan-out happens on the gateway, the obs side already terminates prometheus remote-write (slice 18c). Zero changes on the obs side. The prometheus chart at `infra/k8s-obs/base/prometheus/values.yaml` has been carrying slice-21 TODO comments since slice 17 (lines 6, 10-11, 59-60, 77-80) that need to land or be retracted; this slice does the work the comments named.

## What Changes

- **New DaemonSet `metrics-agent` at `infra/k8s/base/metrics-agent/`** in the `social` namespace, one pod per node, running `otel/opentelemetry-collector-contrib:0.111.0` (same image pin the gateway, the obs collector, and the slice-20 log-agent already share). The pod's config declares:
  - A `kubeletstats` receiver scraping the local node's kubelet at `https://${NODE_NAME}:10250/metrics/resource` and `/stats/summary` every 15s. Uses `auth_type: serviceAccount`, `insecure_skip_verify: true` (k3s kubelet ships a self-signed cert; the agent talks to its own node only, never across nodes). Emits k8s-shaped families: `k8s.node.cpu.*`, `k8s.node.memory.*`, `k8s.pod.cpu.*`, `k8s.pod.memory.*`, `container.cpu.*`, `container.memory.*`, `k8s.volume.*`, plus filesystem/network counters via the `extra_metadata_labels:` block (sets `k8s.persistentvolumeclaim.name` etc. as resource attributes).
  - A `hostmetrics` receiver scraping the node's `/proc`, `/sys`, and root filesystem mounts every 15s. Scrapers enabled: `cpu`, `memory`, `load`, `disk`, `filesystem`, `network`, `paging`, `processes`. Three scrapers (`process`, `processes_temperature`, `system`) are deliberately NOT enabled — they either explode cardinality (`process`) or rely on Linux-specific files not present on every k3s VM image.
  - A `batch` processor and an `otlp` exporter pointing at `collector.social.svc.cluster.local:4317` (gRPC, plaintext, in-cluster).
  - One metrics pipeline: `kubeletstats, hostmetrics → batch → otlp/gateway`. No traces, no logs.
- **New Deployment `metrics-cluster-agent` at `infra/k8s/base/metrics-cluster-agent/`** in the `social` namespace, `replicas: 1` (single-node local cluster has no leader-election story; a future Hetzner slice can revisit), running the same image pin. The pod's config declares:
  - A `k8s_cluster` receiver watching the apiserver. Emits cluster-state families: `k8s.namespace.phase`, `k8s.pod.phase`, `k8s.deployment.desired`, `k8s.deployment.available`, `k8s.replicaset.desired`, `k8s.replicaset.available`, `k8s.statefulset.*`, `k8s.daemonset.*`, `k8s.node.condition_ready`, `k8s.pvc.phase`, `k8s.container.restarts`, and the workload-allocatable resource families.
  - A `batch` processor and the same `otlp` exporter to the gateway.
  - One metrics pipeline: `k8s_cluster → batch → otlp/gateway`.
- **Two ServiceAccount + ClusterRole + ClusterRoleBinding sets**:
  - `metrics-agent` (DaemonSet): cluster-scoped read on `nodes/stats`, `nodes/proxy`, `nodes` (plus `nodes/metrics` for newer kubeletstats builds — kept for forward-compatibility). Required by `kubeletstats` to reach the local kubelet's `https://node:10250`. The hostmetrics receiver itself needs no RBAC — it reads `/proc` and `/sys` from hostPath mounts inside its own pod.
  - `metrics-cluster-agent` (Deployment): cluster-scoped read on the kinds the `k8s_cluster` receiver enumerates per its upstream documentation — `events`, `namespaces`, `namespaces/status`, `nodes`, `nodes/status`, `persistentvolumeclaims`, `persistentvolumes`, `pods`, `pods/status`, `replicationcontrollers`, `replicationcontrollers/status`, `resourcequotas`, `services`, `apps/daemonsets`, `apps/deployments`, `apps/replicasets`, `apps/statefulsets`, `extensions/daemonsets`, `extensions/deployments`, `extensions/replicasets`, `batch/jobs`, `batch/cronjobs`, `autoscaling/horizontalpodautoscalers`. Read-only.
- **DaemonSet pod-spec details for `metrics-agent`**: `hostNetwork: false`, `hostPID: false`, `tolerations: [{operator: Exists}]` (same posture as slice-20 log-agent — pods land on every node including the control-plane-tainted only node). Mounts `hostPath: /` read-only at `/hostfs` for hostmetrics' filesystem and disk scrapers, with `root_path: /hostfs` set in the hostmetrics config. The `NODE_NAME` env var is injected via the downward API (`status.hostIP` is unreliable on multi-CNI cluster setups; `spec.nodeName` is the documented kubeletstats input). Resource requests `cpu=50m, memory=128Mi`; limits `cpu=200m, memory=256Mi` (matches slice-20 log-agent, justified by similar per-record / per-scrape parse + ship work with no cross-cluster TLS).
- **Deployment pod-spec details for `metrics-cluster-agent`**: `replicas: 1`, no tolerations beyond default (no need to schedule everywhere — the receiver watches the apiserver from anywhere it lands), `nodeSelector:` empty. Resource requests `cpu=50m, memory=128Mi`; limits `cpu=200m, memory=256Mi`.
- **Gateway collector ConfigMap is UNCHANGED**: the existing `otlp` receiver already accepts gRPC on `:4317`, the `metrics:` pipeline already dual-writes to `otlphttp/compose-relay-metrics` and `otlphttp/obs-cluster-metrics` (slice 18c, mTLS-wrapped to obs in slice 19). The two new agents plug into that pipeline as additional senders with zero gateway-side change. This is the *whole point* of the agent/gateway pattern slice 20 established.
- **Obs collector ConfigMap is UNCHANGED**: the existing `metrics:` pipeline (batch → `prometheusremotewrite/in-cluster`) already writes into the obs prometheus' remote-write endpoint enabled in slice 18c.
- **Obs prometheus values.yaml at `infra/k8s-obs/base/prometheus/values.yaml`** gets its slice-17 comment block rewritten to reflect what this slice actually does: the four bundled subcharts (`alertmanager`, `prometheus-pushgateway`, `kube-state-metrics`, `prometheus-node-exporter`) all STAY disabled — the OTel-receiver-side approach replaces them at the *agent* layer, not at the prometheus-scrape layer. Every default scrape config under `scrapeConfigs:` STAYS disabled for the same reason (data arrives via remote-write from the obs collector, not via prometheus pull). The comment block at lines 6, 10-11, 59-60, 77-80 is rewritten to drop the "scrape configs land in slice 21" / "kube-state-metrics + node-exporter land in slice 21" hints — those hints predated slice 20's agent/gateway commitment and are now misleading. No live YAML keys change.
- **Obs grafana ConfigMap dashboard provisioning** gains a `cluster-overview.json` dashboard in the same provisioning path the slice-17 chart already wires (provisioning/dashboards). Panels:
  - **Node row**: node CPU usage % per-node; node memory used / available; node load1 / load5 / load15; node disk used % per-mountpoint; node network rx/tx bytes/sec.
  - **Pod row**: per-namespace pod CPU usage; per-namespace pod memory working-set; top-N pods by CPU; top-N pods by memory.
  - **Workload row**: deployment desired vs available replicas (stat panel with thresholds); statefulset / daemonset desired vs available; pod phase distribution stacked bar; container restart count (last 1h, expected 0 — matches compose `infrastructure-overview.json` panel #207).
  - **PVC row**: PVC bound / pending; PVC used %.
  - All panels use Prometheus datasource. PromQL written against OTel-emitted metric names AFTER prometheus-remote-write's dotted-to-underscored translation (e.g. `k8s_node_cpu_utilization`, `k8s_pod_memory_working_set`, `k8s_deployment_available`).
- **justfile gains four recipes**, mirroring the slice-20 `log-agent-*` recipe convention:
  - `metrics-agent-logs`: tail metrics-agent pod logs (follow).
  - `metrics-agent-rollout`: `kubectl rollout restart daemonset/metrics-agent -n social` + wait on `rollout status`.
  - `metrics-cluster-agent-logs`: tail metrics-cluster-agent pod logs (follow).
  - `metrics-cluster-agent-rollout`: `kubectl rollout restart deploy/metrics-cluster-agent -n social` + wait on `rollout status`.
- **Hetzner overlay stub at `infra/k8s/overlays/hetzner/kustomization.yaml`** gains a slice-21 commented block alongside the existing slice-15..20 stubs, naming the production-side concerns: multi-node tolerations, leader election for k8s_cluster on multi-node clusters, resource caps tuned for higher cluster size, optional `prometheus.values.yaml` retention bump alongside cluster-metric volume. Comments only — no live resources.
- **README "Local observability" section** gains a "Cluster metrics" subsection naming the two agents, the apply order (`just k8s-apply` covers it once the base kustomization lists the two new directories), and the expected end-to-end loop (apply → wait one scrape interval → cluster-overview dashboard populates in obs grafana).
- **NON-GOALS** (called out explicitly so a future slice has a clean handoff):
  - No Prometheus Operator / kube-prometheus-stack (README design constraint: each LGTM chart deployed bare, no CRDs).
  - No control-plane metrics (etcd / kube-scheduler / kube-controller-manager). k3s embeds these in the supervisor process and does not expose separate `/metrics` endpoints without `--kube-scheduler-arg=metrics-bind-address=...` flags on the k3s server — out of scope here; a future slice can add them.
  - No alerting rules on cluster metrics. A future alerting slice (post-slice-22) decides which thresholds page.
  - No cardinality / retention tuning. The obs prometheus PVC is 5Gi at 7d retention (chart defaults from slice 17); slice 21 lands the first real metric volume and we measure-then-tune in a later slice.
  - No leader election for `k8s_cluster` receiver. Single-replica Deployment on a single-node local cluster is fine; the Hetzner overlay stub flags this for multi-node prod.
  - No compose-side `cluster-overview` dashboard. The app collector dual-writes metrics, so cluster metrics also reach compose prometheus via the relay — but compose dies in slice 22 (one slice's time); building a compose dashboard now is churn. Operators verifying side-by-side comparability run ad-hoc PromQL in compose grafana's Explore tab during the slice-21 → slice-22 window.
  - No filter / drop list on metric families up front. kubeletstats emits ~60 families, hostmetrics ~30, k8s_cluster ~20 — most are useful, some may be wasteful (per-disk-partition counters on a VM with many virtual disks). Ship everything, observe what hurts the PVC, filter in a later slice.
  - No FE / BE application metric scraping. Slice 6 already pushes those via OTLP; the metrics-agent does not duplicate the path.
  - No scope widening beyond the app cluster. The obs cluster has its own metric-emitting workloads (LGTM stack components); shipping THEIR metrics is the kind of cross-cluster wiring slice 23 (`add-hetzner-deploy`) can decide on once the obs cluster has a stable identity. For now the metrics-agent runs on the app cluster only and the obs cluster keeps its internal-only posture.

## Capabilities

### New Capabilities

None. This slice extends the existing `kubernetes` capability rather than introducing a new one.

### Modified Capabilities

- `kubernetes`: New requirements for the `metrics-agent` DaemonSet, its RBAC, its kubeletstats + hostmetrics pipeline shape, the `metrics-cluster-agent` Deployment, its RBAC, its k8s_cluster pipeline shape, the `cluster-overview` dashboard's end-to-end visibility property, the four justfile recipes, the Hetzner overlay stub, and the README "Cluster metrics" subsection.
- `observability-cluster`: New narrative requirement that the obs prometheus chart values' comment block reflects slice-21 reality (subcharts stay disabled, scrape jobs stay disabled, OTel-receiver-side path replaces the misleading slice-17-era "scrape configs land in slice 21" hints — no chart runtime change). New requirement that the obs grafana chart provisions the `cluster-overview.json` dashboard via the same mechanism the slice-17 `custom-dashboard.json` already uses.

## Impact

- **Code / manifests**: New directories `infra/k8s/base/metrics-agent/` and `infra/k8s/base/metrics-cluster-agent/` (each with `kustomization.yaml`, primary workload manifest, configmap, serviceaccount, rbac). Modified `infra/k8s/base/kustomization.yaml` (`resources:` list gains both new directories). Modified `infra/k8s/overlays/hetzner/kustomization.yaml` (commented slice-21 stub). New `infra/k8s-obs/base/grafana/dashboards/cluster-overview.json` plus a provisioning ConfigMap update if the chart's `dashboardProviders` / `dashboards` blocks need an additional entry. Modified `infra/k8s-obs/base/prometheus/values.yaml` (comment-block rewrite only). Modified `justfile` (four recipes). Modified `README.md` (new subsection).
- **No app-side code change**: backend (`backend/`) and frontend (`frontend/`) are untouched. No image rebuild needed for anything in the social app itself.
- **No new images**: reuses the contrib collector pin already in use by the gateway, the log-agent, and the obs collector.
- **No new cert material**: both new agents → gateway hops are in-cluster plaintext OTLP/gRPC; the slice-19 mTLS envelope stays exactly where it is on the gateway → obs collector hop. The kubeletstats receiver's TLS to the local kubelet uses `insecure_skip_verify: true` (talking to its own node only, k3s ships a self-signed kubelet cert) — no cluster-CA bundling needed.
- **CI impact**: None. CI runs the e2e harness against compose, not against the local k3s cluster.
- **Backwards-compatibility impact**: None. Nothing existing changes behavior; two new producers join the existing pipeline. Compose grafana's `infrastructure-overview.json` (Docker container metrics with `name=` labels) continues to render exactly what it renders today — the slice does not touch it. The Docker-cAdvisor labels (`name`, `image`, etc.) and the k8s labels (`pod`, `namespace`, `container`) are different families, so the new cluster metrics do not collide with or replace the Docker family in compose prometheus; both render side-by-side until slice 22.
- **Resource impact on local Lima VM**: one extra pod (`metrics-agent`) per node + one singleton pod (`metrics-cluster-agent`). At ~256Mi memory limit each = +512Mi memory in the 8 GiB envelope. Fits inside the budget alongside postgres, backend, frontend, gateway, log-agent (current pre-slice headroom ~6 GiB, drops to ~5.5 GiB post-slice).
- **Obs prometheus storage impact**: kubeletstats + hostmetrics + k8s_cluster at 15s scrape interval on a single-node cluster running ~12 pods produces an estimated 2,000-3,000 active series. At 7d retention against a 5Gi PVC this fits comfortably (rule-of-thumb prometheus storage cost is ~1.5 bytes/sample compressed; ~5MB/day for the projected series count). A future Hetzner slice with more nodes / more pods will re-size — flagged in the overlay stub.
- **Backout**: Single revert. Pre-slice state had no cluster-metrics producers; reverting removes both directories + RBAC + base kustomization entries + dashboard JSON + justfile recipes + README subsection. The prometheus values.yaml comment-block rewrite is also reverted but that has no runtime effect. No data migration. The obs prometheus keeps any samples already written (they age out per 7d retention).
