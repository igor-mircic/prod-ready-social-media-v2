## Why

After slice 18b/18c, every backend and browser OTLP signal already fans out from the app k3s cluster to the obs k3s cluster — but the cross-VM hop (app collector → `host.lima.internal:14317/14318` → obs collector receiver) is plaintext OTLP/gRPC and OTLP/HTTP, with `tls.insecure: true` hard-coded on the exporters and no TLS on the receivers. README §"Forward arc" line 455 (and the slice-18b/18c design files, and the slice-18c hetzner-overlay stubs) named this exact slice — slice 19, `add-cross-cluster-mtls` — as the moment the cross-cluster transport becomes authenticated and encrypted. Doing it now, while the local mirror is the only consumer, lets us exercise the trust-distribution primitives (self-signed CA, server cert, client cert) end-to-end before slice 23 has to cope with them on a real two-box Hetzner deploy.

## What Changes

- **New self-signed CA + leaf certs generated locally and partially checked in.** A single self-signed CA is the trust anchor both clusters share. The CA cert (public) is checked in at `infra/observability/certs/ca.crt`; the CA key and all leaf private keys are gitignored and regenerated by a justfile recipe. Two leaf certs: a server cert for the obs collector's OTLP receiver (SANs: `host.lima.internal`, `localhost`, `collector.observability.svc.cluster.local`) and a client cert for the app collector's three `otlphttp/obs-cluster*` exporters.
- **`just obs-certs` recipe** drives the openssl steps end-to-end (CA → server cert → client cert), is idempotent, and is auto-invoked by `just obs-up` if the CA cert is missing. Cert config (CN, SAN extensions, validity) lives in `infra/observability/certs/openssl.cnf` so re-runs are reproducible.
- **Obs collector OTLP receiver requires client cert.** `infra/k8s-obs/base/collector/configmap.yaml` grows `tls:` blocks under `receivers.otlp.protocols.grpc` and `receivers.otlp.protocols.http` declaring `cert_file`, `key_file`, `client_ca_file`, and `require_client_cert: true`. (The pair `client_ca_file` + `require_client_cert: true` is what makes this mTLS rather than one-way TLS. Implementor verifies the exact YAML keys against otelcol-contrib v0.111.0 receiver docs at apply time.)
- **App collector's three obs-cluster exporters become TLS clients.** `infra/k8s/base/collector/configmap.yaml`'s `otlp/obs-cluster`, `otlphttp/obs-cluster-logs`, and `otlphttp/obs-cluster-metrics` exporters lose `tls.insecure: true` and gain `cert_file`, `key_file`, `ca_file`, and `insecure: false`. The OTLP/HTTP exporters' `endpoint:` schemes flip from `http://` to `https://`. The OTLP/gRPC `otlp/obs-cluster` exporter's `endpoint:` stays bare-host:port (gRPC uses scheme-less endpoint with TLS controlled by the `tls:` block). Ports stay `:14317` / `:14318` (TLS is transparent at the Lima portForward L4 level).
- **Compose-relay exporters UNCHANGED.** The three `*compose-relay*` exporters in the app collector's configmap stay plaintext with `tls.insecure: true`. This is a local-only path scheduled for retirement in slice 22; layering mTLS on top of it would be churn-for-zero-benefit.
- **Cert material mounted into both collector pods via Kustomize `secretGenerator`.** Both `infra/k8s/base/collector/kustomization.yaml` and `infra/k8s-obs/base/collector/kustomization.yaml` gain a `secretGenerator` reading the per-cluster `certs/` directory contents into a Secret. Both Deployments mount that Secret at `/etc/otelcol-contrib/certs/` (read-only) alongside the existing config-volume mount. The secretGenerator hashes contents, so regenerating certs triggers a pod restart automatically — no manual rollout needed.
- **Hetzner overlay stubs (`infra/k8s/overlays/hetzner/kustomization.yaml` and `infra/k8s-obs/overlays/hetzner/kustomization.yaml`) gain a one-line note** describing the prod-vs-local diff: same self-signed-CA trust model, but cert distribution moves to cert-manager in slice 23, the CA private key stays off-disk, and the server cert SAN list swaps `host.lima.internal` for the obs box's private-network IP / DNS name. Comments only — no live resources.
- **README "Local observability" section** gains a "Cross-cluster mTLS" subsection naming where certs live, how to regenerate them, and the loud failure mode if certs are missing.
- **NON-GOALS** (called out explicitly so a future slice has a clean handoff): compose-collector leg getting TLS (slice 22 retires it instead); cert-manager adoption (slice 23, with ACME / Let's Encrypt); automated cert rotation (manual `just obs-certs` re-run for now; slice 23 wires cert-manager renewals for production); mTLS on intra-cluster traffic (backend pod ↔ app collector, browser nginx ↔ app collector — same-cluster, separate concern); service-mesh-style mTLS-everywhere (Linkerd / Istio).

- **Affected files / directories:**
  - **New:** `infra/observability/certs/.gitignore` (excludes `*.key` and `*.pem` private-key files); `infra/observability/certs/ca.crt` (checked-in public CA cert); `infra/observability/certs/openssl.cnf` (CA + leaf cert openssl config, reproducible SANs).
  - **New:** `infra/k8s/base/collector/certs/` (app-side client cert + CA cert; private key gitignored, regenerated by the recipe).
  - **New:** `infra/k8s-obs/base/collector/certs/` (obs-side server cert + CA cert; private key gitignored, regenerated by the recipe).
  - **Modified:** `justfile` — new `obs-certs` recipe (openssl steps); existing `obs-up` recipe gains a guard that invokes `obs-certs` if `infra/observability/certs/ca.crt` is missing.
  - **Modified:** `infra/k8s/base/collector/configmap.yaml` — three `otlp/obs-cluster*` exporters gain TLS blocks; OTLP/HTTP endpoint schemes flip to `https://`.

  - **Modified:** `infra/k8s/base/collector/kustomization.yaml` — adds a `secretGenerator` for the app-side cert Secret.
  - **Modified:** `infra/k8s-obs/base/collector/configmap.yaml` — `otlp.protocols.{grpc,http}` receivers gain `tls:` blocks with client-cert verification.
  - **Modified:** `infra/k8s-obs/base/collector/deployment.yaml` — adds a `certs` volume + mount.
  - **Modified:** `infra/k8s-obs/base/collector/kustomization.yaml` — adds a `secretGenerator` for the obs-side cert Secret.
  - **Modified (comments only):** `infra/k8s/overlays/hetzner/kustomization.yaml`, `infra/k8s-obs/overlays/hetzner/kustomization.yaml`.
  - **Modified:** `README.md` — "Cross-cluster mTLS" subsection under "Local observability".
  - **Modified:** `.gitignore` — pattern for `infra/**/certs/*.key` (defence-in-depth; per-directory `.gitignore` also covers it).
- **New tool dependencies:** `openssl` (already present on macOS / Lima images by default; the recipe asserts presence and fails loudly with a hint if absent).
- **Backwards-compatibility impact:** A developer running `just obs-up` on an existing checkout without re-running cert-gen will hit a TLS handshake failure on the receiver side (obs collector readiness probe still passes — the failure surfaces in the app collector's exporter retry log). The `obs-up` guard makes this a non-issue for the fresh-clone path; the manual-edit path is documented in the README. The compose collector path is unaffected: backend traces continue to land in compose grafana on `:3000` exactly as before, since the compose-relay leg of the dual-write is unchanged.
- **CI impact:** None. CI does not bring up the obs Lima VM; the e2e harness runs against the app cluster's compose-relay path. The cert material is local-developer-only until slice 23.
- **Backout:** Single revert. Pre-slice-19 state is plaintext on the cross-VM hop; reverting this slice restores `tls.insecure: true` on the three exporters and removes the receiver-side TLS blocks. No data migration, no schema change.
