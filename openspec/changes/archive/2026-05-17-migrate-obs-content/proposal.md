## Why

Slices 17–21 finished building the *transport spine* of the two-cluster observability migration: the app-cluster gateway dual-writes traces/logs/metrics to the obs cluster under mTLS (slices 18b/18c/19), and the obs cluster now carries the only two pod-side producers (`log-agent` slice 20, `metrics-agent` + `metrics-cluster-agent` slice 21). What the obs side is *missing* is **content** — the SLO + database alerting rules, the alertmanager routing tree (page/ticket webhook receivers + BackendDown inhibition), the backend/frontend/database dashboards, and the two underlying signal sources that compose was carrying that the slice-21 receivers do not cover (postgres-internal metrics via `postgres-exporter`; the webhook-sink mock used by the alerting end-to-end test).

The README's "Forward arc" (line 442) queues this as part of slice 22 `retire-compose-observability`, but the actual work behind that name is two units of work joined at the hip:

1. **Migrate** content from compose into the obs cluster, with compose still running so the operator can render the same dashboards on `:3000` and `:3001` side-by-side and confirm parity.
2. **Retire** the compose stack — drop the `compose-relay*` exporters from the app collector, delete the `observability` compose profile, retarget the e2e specs, rename `infra/observability/`.

Doing both in one slice throws away the side-by-side parity window exactly when it's most useful — rule and dashboard translation is the riskiest content to move, and the arc's stated invariant (README line 446-448) is *"each slice is independently revertable and the visibility-into-the-app-cluster property is never broken."* Slice 22a does the migration; slice 22b (next) does the retirement. Each is independently revertable. The dual-write spine introduced in slices 18b–21 stays intact through 22a and only collapses in 22b.

This split also lets 22a stay narrowly content-focused (no app-collector ConfigMap re-shape, no e2e retarget, no README narrative rewrite). 22b is then a delete-only slice with a small set of port-forward rewires.

## What Changes

- **Obs prometheus rule files at `infra/k8s-obs/base/prometheus/rules/`** — new directory carrying copies of `slo-recording.yml`, `slo-alerting.yml`, `fe-slo-recording.yml`, `fe-slo-alerting.yml`, `database-alerts.yml` (read directly from `infra/observability/prometheus/rules/` — the compose-side copies stay where they are as the parity reference). Mounted into the obs prometheus pod via a `configMapGenerator` in `infra/k8s-obs/base/prometheus/kustomization.yaml` producing `ConfigMap/prometheus-extra-rules`, surfaced to the chart via `extraConfigmapMounts:` pointing at `/etc/prometheus-extra-rules/`. The chart's existing `serverFiles.prometheus.yml.rule_files:` entry list gains the extra mount path. Container-alerts (`container-alerts.yml`) is *not* migrated — it's keyed on cadvisor-shaped series (`container_cpu_cfs_throttled_periods_total`, `container_memory_working_set_bytes`, `name=` / `image=` labels) that don't exist in the OTel-shaped slice-21 families. Rewriting it is real spec evolution, not content migration; flagged as a follow-up.
- **Obs prometheus `server.alertmanagers:` wired to the in-cluster alertmanager**. `infra/k8s-obs/base/prometheus/values.yaml` gains a `server.alertmanagers:` list with a single `static_configs:` target of `alertmanager.observability.svc.cluster.local:9093`. This is the leg the slice-17 obs prometheus was missing — `rule_files:` could be set but firings had no receiver.
- **Obs alertmanager config replaces the slice-17 null receiver** with the routing tree migrated from `infra/observability/alertmanager/alertmanager.yml`. `infra/k8s-obs/base/alertmanager/values.yaml` gains: a top-level `route:` block keying on `severity="page"` → `page-webhook` and `severity="ticket"` → `ticket-webhook`, the same `group_by`/`group_wait`/`group_interval`/`repeat_interval` values, a `BackendDown` inhibition rule, and three receivers (`default`, `page-webhook`, `ticket-webhook`) whose webhook URLs target the in-cluster `webhook-sink` Service.
- **`webhook-sink` Deployment + Service at `infra/k8s-obs/base/webhook-sink/`** in the `observability` namespace. Source image is built from `infra/observability/webhook-sink/` (already exists for compose), tagged `webhook-sink:dev` and pushed to the local OCI registry (`registry.local:5000/webhook-sink:dev`) under the same path the slice-15 backend/frontend images use. The Service exposes ClusterIP on port `8080`; the receivers in alertmanager reach it at `http://webhook-sink.observability.svc.cluster.local:8080/page` and `/ticket`. The pod's `/received` query surface stays on `:8080`; an in-cluster operator can `kubectl port-forward` it for the alerting e2e test (slice 22b decides whether to keep that test).
- **`postgres-exporter` Deployment + Service + ServiceAccount at `infra/k8s/base/postgres-exporter/`** in the `social` namespace. The exporter image (`quay.io/prometheuscommunity/postgres-exporter:v0.17.1`, same pin as compose) reaches the in-cluster postgres at `postgres.social.svc.cluster.local:5432`, projects the `pg_stat_statements` custom-queries file from `infra/observability/postgres-exporter/queries.yaml` via a `configMapGenerator`. Pod env loads `DATA_SOURCE_USER` / `DATA_SOURCE_PASS` / `DATA_SOURCE_URI` from the same `postgres-credentials` Secret the backend uses (slice 14 source of truth). The Service exposes ClusterIP on `:9187`.
- **App collector ConfigMap gains a `prometheus` receiver** scraping `postgres-exporter.social.svc.cluster.local:9187/metrics` every 15s. The receiver's metric stream joins the existing kubeletstats/hostmetrics/k8s_cluster traffic at the `batch` processor, then through the existing fan-out to both `prometheusremotewrite/in-cluster` (obs) and `otlphttp/compose-relay-metrics` (compose). Both prometheus instances see `pg_*` series under the same names slice-12 declared, validating parity. No new pipeline; the existing `metrics:` pipeline gains one receiver.
- **Three migrated grafana dashboards at `infra/k8s-obs/base/grafana/dashboards/`** — `backend-overview.json`, `frontend-overview.json`, `database-overview.json` copied from `infra/observability/grafana/dashboards/` with two surgical edits: (i) datasource UIDs flipped from compose's `prometheus` UID to the obs UID (already `prometheus` per slice-18b naming, so the JSON is byte-identical for this leg); (ii) any `instance="host.docker.internal:8080"` selector relaxed to `instance=~".*"` because the BE in k3s exports under a different `instance` label. The compose `infrastructure-overview.json` is *not* migrated — slice 21 already produced `cluster-overview.json` covering the same operator role with k8s-shaped families.
- **Obs grafana ConfigMap dashboard provisioning** picks up the three new JSON files via the same mechanism the slice-21 `cluster-overview.json` already uses (the chart's `dashboards:` block under `dashboardProviders:`). No new provisioning shape; three more entries.
- **CI rule-file validation extended**. The existing `prometheus-rules` job in `.github/workflows/ci.yml` (`promtool check rules` + `promtool test rules`) continues to validate the compose-side files. A new job step asserts the obs-side copies stay byte-identical via `diff -q infra/observability/prometheus/rules/<file>.yml infra/k8s-obs/base/prometheus/rules/<file>.yml` so drift between the parity-window copies is caught at PR time. Slice 22b drops the compose-side files and the diff step.
- **README "Local observability cluster" subsection gains a "Migrated content" paragraph** naming what's now in obs prom / obs grafana / obs alertmanager and the parity window. The compose section explicitly stays — slice 22a does not retire compose.
- **justfile gains one recipe**: `obs-webhook-sink-received` — `kubectl --context obs -n observability exec -it deploy/webhook-sink -- curl -s localhost:8080/received | jq` — so the alerting parity check is one command away. Mirrors the compose-side `just webhook-received` recipe if one exists; if not, this is the first.
- **Hetzner overlay stub at `infra/k8s/overlays/hetzner/kustomization.yaml`** gains a slice-22a commented block flagging: postgres-exporter as a per-cluster sidecar in prod (not centralized), separate `pg_monitor`-granted DB role for the exporter (no shared application credentials), and resource-cap re-sizing if many namespaces are observed.

## Capabilities

### New Capabilities

None. This slice extends `kubernetes` and `observability-cluster`.

### Modified Capabilities

- `kubernetes`: new requirements for the `postgres-exporter` Deployment + Service + ServiceAccount in the `social` namespace, the kustomize ConfigMap for the custom-queries file, and the app collector's new `prometheus` receiver scraping it.
- `observability-cluster`: new requirements for the obs prometheus extra rules ConfigMap + chart wiring + alertmanager target; the obs alertmanager routing tree replacing the slice-17 null receiver; the `webhook-sink` Deployment + Service; and provisioning of the three migrated grafana dashboards.

## Impact

- **Code / manifests**: new directories `infra/k8s/base/postgres-exporter/`, `infra/k8s-obs/base/prometheus/rules/`, `infra/k8s-obs/base/webhook-sink/`. Modified `infra/k8s/base/kustomization.yaml`, `infra/k8s-obs/base/kustomization.yaml`, `infra/k8s/base/collector/configmap.yaml` (one receiver added), `infra/k8s-obs/base/prometheus/values.yaml` (alertmanagers target + extra rules mount), `infra/k8s-obs/base/prometheus/kustomization.yaml` (configMapGenerator for rules), `infra/k8s-obs/base/alertmanager/values.yaml` (full config replacement), `infra/k8s-obs/base/grafana/values.yaml` (three dashboard provisioning entries), `infra/k8s/overlays/hetzner/kustomization.yaml` (commented stub), `justfile` (one recipe), `README.md` (one paragraph), `.github/workflows/ci.yml` (one diff step).
- **No app-side code change**: backend (`backend/`) and frontend (`frontend/`) are untouched.
- **No images**: postgres-exporter image reused from compose pin; webhook-sink built from the existing `infra/observability/webhook-sink/` sources and pushed to the local registry under the same flow slice 15 established for backend/frontend.
- **No new cert material**: postgres-exporter → app collector is in-cluster plaintext; collector → obs prom is the existing slice-19 mTLS leg. The webhook-sink is alertmanager → ClusterIP plaintext (in-cluster), not cross-cluster.
- **CI impact**: one new diff step in the `prometheus-rules` job (~2s). Promtool runs against the compose-side rule files unchanged.
- **Backwards-compatibility impact**: none. Compose observability keeps running exactly as it does today. Both prometheus instances now scrape `pg_*` series; their alertmanagers both fire the same SLO rules; their grafanas render the same backend/frontend/database dashboards. Slice 22a is purely additive.
- **Resource impact on local Lima VMs**: app cluster gains one Deployment (postgres-exporter, ~64Mi); obs cluster gains one Deployment (webhook-sink, ~32Mi) plus the chart-templated rules ConfigMap and one extra mount on the existing prometheus pod. Total ~+128Mi across the two VMs.
- **Obs prometheus storage impact**: `pg_*` family adds ~20-30 active series, fits in the existing 5Gi PVC at 7d retention without re-sizing.
- **Side-by-side parity window**: an operator running compose + obs side-by-side sees the same series names, the same alert firings (in two alertmanagers; one is the source of truth, the other is the candidate replacement), and the same dashboard renders on `:3000` and `:3001`. The alerting e2e test (`e2e/tests/observability.alerting.spec.ts`) keeps targeting compose's webhook-sink on `:8081`; an operator can manually port-forward the obs webhook-sink for the same check. Slice 22b decides whether the e2e retargets or is retired.
- **Backout**: single revert. Pre-slice state had no rules / alertmanager routing / postgres-exporter / webhook-sink / migrated dashboards on the obs side; reverting removes all of them. The chart values changes back out cleanly. No data migration. Compose remains the source of truth throughout.
